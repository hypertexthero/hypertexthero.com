REST Architecture Notes

Friday, February 15 2013

Don’t remember what I was reading when I wrote these down. For posterity, notes on REpresentational State Transfer.

  • Architecture style, not toolkit
  • Distillation of the way the web already works
  • Resources are identified by uniform resource identifiers (URIs, or URLS):
    domain.tld/thing/12/
  • Resources are manipulated through their representations:
    domain.tld/thing/12/edit/
  • Messages are self-descriptive and stateless
  • Multiple representations are accepted or sent
  • Hypertext is the engine of application state

VERBS (HTTP Request Methods)

GET
POST
PUT
DELETE

NOUNS (URIs)

domain.tld/

Why Hypertext?

  1. Because the links mirror the structure of how a user makes progress through an application
  2. The user is in control, thanks to the back button and other non-local actions
  3. In a web service, the client should be in control in the same sense

Web-based Applications

A web-based application is a dynamically changing graph of:

  1. State representations (pages)
  2. Potential transitions (links) between states

If it doesn’t work like that, it may be accessible from the web, but it’s not really part of the web.

Advantages of REST Architecture

1. Architectural constraints when applied as a whole, generate:

  • Scalable component interactions
  • General interfaces
  • Independently deployed connectors
  • Reduced interaction latency
  • Strengthened security
  • Safe encapsulation of legacy systems

2. Separates server implementation from clients’ perception of resources (cool URIs don’t change).

3. Enables transfer of data in streams of unlimited size and type.

Some Data Elements

Resources: The intended conceptual target of a hypertext reference
Resource identifiers: URIs
Rsource metadata: Source links, alternates
Representations: HTML documents, JPEG images
Representation-specific metadata: media-type, last-modified time

Best Argument for using REST?

A serviced offered in a REST style will inherently be easier to use than some complex API because:

  1. Lower learning curve for consumer/user
  2. Lower support overhead for producer/maintainer

Three Hypotheses of Human Interface Design :

  1. Minimize number of text fields to absolute minimum
  2. Minimize number of clicks/keystrokes/gestures necessary to accomplish actions
  3. Make interface as responsive as possible. Minimize latency of each action user might take.1

1 XmlHttpRequest object lets you do REST from inside a web page (AJAX): An API available in web browser. Scripting languages like JavaScript. Used to send HTTP or HTTPS request directly to web server and load the server response data directly into the script.
